## 🛠️ 自修复流程

### 完整流程

```
1. 发现问题（监控）
   ↓
2. 定位根因（诊断）
   ↓
3. 自动修复（自修复）
   ↓
4. 验证修复（验证）
   ↓
5. 记录学习（记忆）
```

### 示例：止损单问题

**发现**:
```python
# 检查持仓和止损单
positions = get_positions()
stop_orders = get_stop_orders()

for pos in positions:
    if pos['size'] > 0 and not has_stop_order(pos):
        issue = f"{pos['symbol']}: 有持仓但无止损单"
```

**修复**:
```python
# 自动创建止损单
create_stop_loss(
    symbol=pos['symbol'],
    size=pos['size'],
    stop_price=current_price * 0.98
)
```

**验证**:
```python
# 再次检查
stop_orders = get_stop_orders()
if has_stop_order(pos):
    log("✅ 已修复")
```

**记录**:
```python
# 记录到日志和 MEMORY.md
log("修复止损单问题")
```

## 📋 监控规则

### 告警规则

| 问题 | 告警条件 | 冷却时间 |
|------|---------|---------|
| 进程异常 | 立即告警 | 5 分钟 |
| 持仓无止损 | 立即修复，修复失败告警 | 5 分钟 |
| API 失败 | 连续 3 次失败 | 5 分钟 |
| 余额不足 | 低于 100 USDT | 30 分钟 |

### 自修复规则

| 问题 | 自修复动作 | 验证方式 |
|------|-----------|---------|
| 进程挂了 | 自动重启 | 检查进程状态 |
| 持仓无止损 | 自动创建止损单 | 检查止损单 |
| API 失败 | 自动重试（3 次） | 检查 API 响应 |

## 🧠 核心教训

### 之前的监控缺陷

| 缺陷 | 后果 | 改进 |
|------|------|------|
| 只监控进程 | 发现不了业务问题 | 添加业务逻辑监控 |
| 只告警不修复 | 依赖用户手动 | 添加自修复 |
| 没有验证 | 不知道修复成功 | 添加验证步骤 |
| 没有记忆 | 重复犯错 | 记录到 MEMORY.md |

---

### 正确的监控设计

1. **多层次监控** - 进程 + 业务 + 性能
2. **自动修复** - 能自动就不手动
3. **验证闭环** - 修复后验证
4. **记录学习** - 避免重复犯错
